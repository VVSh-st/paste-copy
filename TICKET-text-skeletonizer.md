# Тикет: TextSkeletonizer — сжатие текста для LLM-запросов

## Роль
Ты — аудитор по архитектуре LLM-интеграций. Твоя задача — проанализировать предложенное решение и дать рекомендации.

**ВАЖНО: Начни ответ с представления — кто ты (роль/экспертиза), чтобы мы могли оценить компетентность.**

## Контекст

Приложение paste\copy — веб-приложение для работы с текстовыми промптами. Есть LLM-интеграция (flowchart, mindmap, тезаурус, рерайт, аудит и др.).

Проблема: LLM получает сырые тексты до 100K+ символов. Это:
- Дорого (токены = деньги)
- Медленно
- LLM теряется в большом объёме → хуже результат

## Решение

Создан `TextSkeletonizer` — компактный класс (~200 строк), который извлекает структуру из текста:

```
Вход: "Интервью с кандидатом" (20K символов)
Выход:
=== ДОКУМЕНТ ===
Интервью с кандидатом на позицию Frontend Developer

=== СТРУКТУРА ===
# Введение
  Кандидат имеет 5 лет опыта в веб-разработке
# Опыт работы
  Работал в компаниях A, B, C
  Основные проекты:电商平台, CRM-система
# Технические навыки
  React, TypeScript, Node.js
# Вопросы
  1. Расскажите о опыте с React
  2. Как вы подходите к тестированию?

=== КЛЮЧЕВЫЕ ПОНЯТИЯ ===
react, typescript, node.js, тестирование, компоненты

=== СПИСКИ ===
[Основные проекты]:
  - E-commerce платформа
  - CRM-система
  - Дашборд аналитики

=== СТАТИСТИКА ===
19.5 KB | 3200 слов | 12 абзацев | 6 секций
```

## Вопросы для аудитора

1. **Архитектура**: Правильный ли подход — делать сжатие на клиенте перед отправкой в LLM? Или лучше отправлять сырой текст и пусть LLM сам решает что важно?

2. **Порог сжатия**: 8000 символов — правильный порог? Стоит ли делать адаптивный (зависит от модели/провайдера)?

3. **Качество сжатия**: Какие методы извлечения структуры лучше для русского языка? Нужен ли NLP (стемминг, лемматизация) или хватает эвристик?

4. **Настройки**: Стоит ли добавить degree of compression (light/medium/aggressive)? Какие параметры должны быть доступны пользователю?

5. **Кэширование**: Стоит ли кэшировать скелеты (hash от текста → скелет)?

6. **Альтернативы**: Есть ли лучше подходы? Например:
   - Двухэтапный: LLM извлекает структуру → по структуре строим схему
   - Map-reduce: разбиваем на чанки → каждый анализируем → объединяем
   - Embeddings: ищем релевантные фрагменты через векторный поиск

7. **Ошибки**: Какие подводные камни могут быть при сжатии текста?
