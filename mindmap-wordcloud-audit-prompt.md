# Задание для LLM-аудитора: `mindmap.js`

Работаешь с проектом `E:\CODE\Paste_copy`, файл: `mindmap.js`.

Нужно проверить и улучшить реализацию MindMap без расползания по проекту. Меняй минимум файлов, предпочтительно только `mindmap.js`. Не добавляй зависимости. Не делай новый фреймворк/архитектуру. Исправляй причину в общем месте.

## Цель

1. Облако слов должно быть реальным локальным word cloud из текста preview:
   - считать слова кодом на клиенте;
   - показывать сразу при открытии mindmap, пока LLM-анализ еще выполняется;
   - сохранять красивую SVG-визуализацию: разные размеры, цвета, аккуратное размещение;
   - не запрашивать и не использовать `words` от LLM для режима облака слов.

2. Улучшить переход по слову при клике:
   - если слово встречается в тексте несколько раз, переход должен выбирать понятное/лучшее вхождение, а не случайно первое неудачное;
   - поведение должно быть предсказуемым для пользователя и не ломать текущий `mindmap:jump-word` контракт.

## Текущий контекст

- `open()` сейчас открывает overlay и, если `_data` пустая, запускает `_fetch(text)`.
- `_fetch(text)` вызывает `window.LLMCore?.request?.()` и нормализует JSON через `_normalizeData`.
- `_normalizeData(raw)` сейчас принимает `data.words` от LLM и использует их для `words`, `links`, `clusters`.
- `_drawWords(W, H)` рисует облако из `_data.words`, а частотность досчитывает через `_countOccurrences(sourceText, item.w)` по `Preview.getText().slice(0, 4000)`.
- `_jumpToWord(word)` отправляет `CustomEvent('mindmap:jump-word', { detail: { word, exact } })`.

## Требование 1: локальное облако слов

Сделай локальный расчет слов отдельной маленькой функцией рядом с word-cloud кодом.

Минимальная ожидаемая логика:
- брать текст из `window.Preview?.getText?.() ?? ''`;
- токенизировать RU/EN слова через Unicode-regex;
- убрать мусор: короткие слова, частые RU/EN стоп-слова, чистые числа, слишком длинные токены;
- считать частоты по всему доступному preview или по разумному лимиту, если полный текст может быть огромным;
- вернуть максимум 60-80 слов;
- вес слова считать из частоты, например `1..10`;
- сохранить данные в формате, совместимом с текущим `_drawWords`: `{ w, weight, role, count }`.

Важно:
- LLM больше не должен быть источником слов для word cloud.
- В LLM prompt/request не проси `words` для облака. Если `words` нужны только для графа, либо убери их из prompt, либо явно отдели: локальное облако не зависит от LLM words.
- При открытии `open()` сначала рисуй локальные words сразу, затем параллельно запускай `_fetch(text)` для graph/tree/clusters/hierarchy/timeline.
- Когда LLM ответит, не затирай локальное облако слов. Режим `words` должен продолжать показывать локальные слова.
- Если пользователь нажал refresh, локальное облако должно пересчитаться сразу из актуального preview.
- Если текст пустой, старое поведение с toast сохранить.

Проверь связанные режимы:
- `graph` и `links` сейчас завязаны на `_data.words`. Если убираешь LLM words, не сломай граф. Самый простой рабочий путь: использовать локальные words как общий список узлов, а LLM пусть возвращает связи/кластеры/иерархию/шаги. Линки фильтровать по локальным словам.
- Если LLM вернул связи к словам не из локального топа, не падай. Либо мягко отфильтруй, либо добавь недостающие слова ограниченно и пометь их малым весом, но не показывай их в word cloud как основные.

## Требование 2: лучший прыжок по слову

Сейчас событие передает только `word` и `exact`. Этого мало, если вхождений 5.

Нужно улучшить выбор вхождения без большой системы:
- вычислять для слова список позиций в исходном тексте;
- передавать в `detail` дополнительную подсказку, совместимую со старым контрактом: например `index`, `occurrence`, `positions` или `context`;
- старые поля `word` и `exact` не удалять;
- если есть несколько вхождений, выбирать наиболее полезное:
  - предпочтительно exact word boundary;
  - не выбирать часть другого слова;
  - можно выбрать вхождение ближе к началу основного текста, но лучше избегать заголовков/повторов, если рядом нет осмысленного контекста;
  - для одинаковых вариантов допускается простая стабильная стратегия: первое exact-вхождение с коротким контекстом.

Не усложняй:
- не нужна новая UI-панель выбора вхождения;
- не нужен индексатор всего документа;
- не нужен fuzzy search, если exact word boundary работает.

## Acceptance criteria

- При первом открытии mindmap режим `words` появляется сразу, до завершения LLM-запроса.
- Если LLM недоступен или вернул ошибку, локальное облако слов все равно остается видимым.
- Визуально слова имеют разные размеры и цвета, частые слова заметно крупнее редких.
- LLM prompt больше не требует слов для облака слов.
- LLM response не может затереть локальное облако мусорными `words`.
- Refresh пересчитывает локальное облако из текущего текста.
- Клик по слову отправляет `mindmap:jump-word` со старым `word` и дополнительной подсказкой для выбора конкретного вхождения.
- Для слова с 5 вхождениями выбирается стабильное exact-вхождение, не подстрока внутри другого слова.
- Пустой preview не открывает overlay и показывает toast, как раньше.

## Риски, которые надо проверить

- `_normalizeData` не должна падать, если LLM больше не возвращает `words`.
- `_drawGraph` не должен падать на пустых/отфильтрованных links.
- `_drawWords` не должен делать тяжелый пересчет на каждый render, если это заметно тормозит. Достаточно хранить локально рассчитанные words в `_data`.
- Не сломать режимы `tree`, `clusters`, `hierarchy`, `timeline`.
- Не добавлять новые глобальные listeners.
