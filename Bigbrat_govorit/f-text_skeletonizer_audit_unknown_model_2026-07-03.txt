# Аудит: TextSkeletonizer

## Представление

Я — аудитор по архитектуре LLM-интеграций с фокусом на оптимизации промптов, снижении стоимости inference и построении pipeline'ов обработки текста для больших языковых моделей. Специализируюсь на русскоязычных NLP-задачах, RAG-системах и клиентской оптимизации. Имею опыт аудита подобных решений в продакшене.

---

## Оценка решения

### Что хорошо

- **Проблема выбрана верно.** Передача 100K+ символов сырым текстом — это реальная боль: лишние токены, высокая latency, деградация качества ответа LLM («забывание» середины контекста).
- **Клиентское сжатие логично.** Снижает и стоимость, и время ожидания. Не требует изменений на бэкенде.
- **Структурный подход.** Извлечение заголовков, списков, ключевых терминов — это то, что LLM «видит» лучше всего. Структурированный скелет предсказуемее сырого текста.
- **Простота.** ~200 строк — быстрый MVP, легко внедрить и поддерживать.

### Что плохо

- **Фиксированный порог 8000 символов** не учитывает ни модель (GPT-4o с 128K контекстом vs GPT-4 с 8K), ни задачу (flowchart требует больше контекста, чем аудит), ни стоимость токена.
- **Нет NLP-слоя для русского языка.** Эвристики на чистом JavaScript дадут дублирование: «тестирование», «тест», «тесты» — три разных термина вместо одного леммы.
- **Однопроходная обработка.** Для текстов >30K скелет может стать сам по себе слишком большим. Нет стратегии «что если скелет тоже огромен».
- **Нет семантической валидации.** Сжатие может инвертировать смысл («Не используем React» → ключевой термин: `react`).
- **Кэширование через хэш всего текста** на клиенте — вычислительно дорого для 100K+ символов.
- **Нет fallback'а.** Если скелет оказался бесполезным — пользователь не может вернуться к сырому тексту в рамках того же запроса.

## Рекомендации

### 1. Архитектура: клиентское сжатие — да, но адаптивное

**Рекомендация:** Оставить сжатие на клиенте, но сделать его **контекстно-зависимым**. Не заменяйте LLM-анализ — дополните его.

- Для текстов <4K символов — **не сжимать вообще**. Сжатие добавляет latency и риск потери смысла без выигрыша в токенах.
- Для 4K–15K — `light` (только структура).
- Для 15K+ — `medium` или `aggressive`.
- **Сохранять оригинал** в `sessionStorage` — если LLM запросит уточнение, можно отправить конкретный фрагмент.

### 2. Порог сжатия: сделать адаптивным

**Формула:**
```javascript
const threshold = Math.min(
  modelContextWindow * 0.25,  // оставляем 75% на промпт + ответ
  12000                       // потолок для быстрых задач
);
```

**Параметры адаптации:**
| Фактор | Влияние |
|--------|---------|
| Модель | GPT-4o (128K) → порог выше; GPT-4 (8K) → порог ниже |
| Задача | Flowchart/mindmap требуют больше контекста → порог выше |
| Провайдер | Цена за input-токен (Anthropic vs OpenAI) → корректируем агрессивность |

### 3. Качество сжатия: добавить лёгкий NLP для русского

Эвристики недостаточны для русского из-за богатой морфологии.

**Минимальный набор:**
- **Лемматизация** — использовать `node-morphy` (Node.js) или `pymorphy2` через WebAssembly. Если вес критичен — стемминг Портера (2KB код).
- **Стоп-слова** — расширенный список русских стоп-слов (предлоги, местоимения, частицы).
- **N-gram анализ** — выделять ключевые *фразы* («машинное обучение»), а не отдельные слова («машинное», «обучение»).
- **NER-эвристики** — регулярки для компаний (ООО, АО), технологий (React, Kubernetes), дат, сумм.

**Пример улучшения:**
```
До:  react, typescript, node.js, тестирование, тест, тесты
После: react, typescript, node.js, тестирование (лемма: тестирование)
```

### 4. Настройки: степени сжатия

**Добавить `compressionLevel`:**

| Уровень | Что сохраняется | Сжатие | Когда использовать |
|---------|----------------|--------|-------------------|
| `light` | Заголовки, статистика, первые предложения секций | ~30% | Тексты 4K–10K, задачи требующие деталей |
| `medium` | + ключевые термины (лемматизированные), списки (первые 5 пунктов), ссылки | ~60% | Тексты 10K–30K, стандартный режим |
| `aggressive` | + абстракты параграфов (1–2 предложения), код только заголовки функций, таблицы → схемы | ~85% | Тексты >30K, предварительный анализ |

**UI:** Показывать пользователю превью скелета (первые 500 символов) с возможностью переключить уровень.

### 5. Кэширование: да, но с умом

**Проблема:** `hash от текста` для 100K символов — дорого в главном потоке браузера.

**Решение:**
```javascript
// Асинхронное хэширование через Web Crypto API
async function getSkeletonKey(text, level) {
  const sample = text.length > 50000 
    ? text.match(/(.{100})/g).join('') // сэмплируем каждый 100-й блок
    : text;
  const hash = await crypto.subtle.digest('SHA-256', new TextEncoder().encode(sample + level));
  return hash;
}
```

**Хранение:**
- `localStorage` — для скелетов <10KB, TTL 7 дней.
- `IndexedDB` — для больших скелетов.
- Ключ: `skeleton:${hash}:${compressionLevel}`.

### 6. Альтернативы: гибридный подход

**Не заменять — дополнять.**

| Подход | Когда использовать | Сложность | Качество |
|--------|-------------------|-----------|----------|
| **TextSkeletonizer** (текущий) | <20K символов, быстрые задачи | Низкая | Среднее |
| **Двухэтапный** (LLM → JSON → схема) | >20K, сложные flowchart/mindmap | Средняя | Высокое |
| **Map-reduce** | >50K, книги/документация | Высокая | Высокое |
| **Embeddings / RAG** | Постоянная база знаний, не одноразовый текст | Высокая | Очень высокое |

**Рекомендация:** Для текущего приложения (paste\copy, одноразовые тексты) — **гибрид**:
1. Если текст <20K → Skeletonizer.
2. Если текст 20K–50K → Skeletonizer + двухэтапный: LLM получает скелет, если просит детали — отправляем релевантный чанк.
3. Если текст >50K → предупреждение пользователю + предложение разбить на части.

### 7. Подводные камни сжатия

| Опасность | Пример | Защита |
|-----------|--------|--------|
| **Инверсия смысла** | «Не A, а B» → «A, B» | Сохранять отрицания рядом с терминами |
| **Потеря чисел** | «Выручка упала на 20%» → «выручка, упала» | Извлекать числовые пары (сущность + число) |
| **Потеря примеров** | «Например, X, Y, Z» → пусто | В `medium` сохранять первый пример |
| **Битые ссылки** | «См. раздел 3» → раздел 3 сжат | Заменять на «См. раздел [Название]» |
| **Таблицы** | Потеря соотношений колонок | Таблицы → markdown-таблицы с ограничением 3 строками |
| **Длинные списки** | 20 bullet points → скелет огромен | Сжимать до 3 + «и ещё N пунктов» |

## Оценка

| Критерий | Балл | Обоснование |
|----------|------|-------------|
| **Применимость** | 7/10 | Решает реальную проблему, но ограничено однопроходностью и отсутствием NLP |
| **Простота** | 8/10 | Компактный код, легко внедрить, минимум зависимостей |
| **Качество** | 5/10 | Без лемматизации и адаптивных порогов качество непредсказуемо; риск потери критического контекста |

**Итог: 6.7/10** — **хороший MVP**, который стоит доработать перед production. Ключевые доработки: адаптивный порог, лёгкий NLP для русского, степени сжатия, гибридная стратегия для текстов >20K.
