# Аудит llm-core.js

## Правила аудита

1. **Назови свою модель** — первая строка ответа: `Я — <название модели>`
2. **Формат тикетов** — строго по шаблону:

```
[уровень] Краткое название
- Строка: <номер/диапазон>
- Описание: <что не так>
- Влияние: <чем грозит>
- Рекомендация: <конкретный код>
```

3. **Уровни**: critical / high / medium / low
4. **Что искать**: баги, утечки памяти, race conditions, логические ошибки, edge cases, проблемы с сетевыми запросами
5. **Что НЕ искать**: стиль кода, именование переменных, комментарии, отступы
6. **Опасные улучшения** — не предлагай: переписывание архитектуры, добавление зависимостей, смена API

## Контекст проекта

Vanilla JS, localStorage, веб-редактор промптов. `llm-core.js` (~1900 строк) — ядро LLM-интеграции:

**Основные компоненты:**
- `PROVIDERS` — реестр провайдеров (LM Studio, Ollama, OpenAI, Groq, Cherry Studio, OpenRouter)
- `FetchSemaphore` — очередь запросов (max 3, + отдельная для локальных провайдеров max 1)
- `LLMCache` — кэш ответов через localStorage (TTL, LRU)
- `LLMRequestLog` — история запросов через localStorage
- `request()` — основная функция: fetch с retry, timeout, streaming (SSE/NDJSON), abort, кэширование
- `loadModels()` — загрузка списка моделей (OpenAI, Ollama, Cherry с пагинацией)
- `_parseSSE()` / `_parseNDJSON()` — парсеры стриминга
- `_buildRequestBody()` — формирование тела запроса (chat/completions + responses API)
- `LLMSettingsModal` — UI модального окна настроек ( ~1100 строк)
- `BUILTIN_PROMPTS` — 30+ встроенных промптов для различных функций

**Паттерны:**
- Все запросы через `fetch` с `AbortController` + `setTimeout` для таймаута
- SSE парсит `data:` строки, NDJSON — построчно
- Retry на 429 и пустой ответ ( exponential backoff)
- `_withNoThink` — инъекция `/no_think` для локальных моделей
- `_esc()` — XSS-экранирование для HTML-вывода в `LLMSettingsModal`

## Чек-лист

### Network / Request
- [ ] `request()`: корректно ли работает retry на 429? `retries` декрементируется правильно?
- [ ] `request()`: что будет при `AbortError` + `timedOut` одновременно?
- [ ] `request()`: `opts.signal` + внутренний `controller` — нет ли двойного abort?
- [ ] `request()`: `finally` блок — гарантированно ли `sem.release()` вызывается при任何 исключении?
- [ ] `_buildHeaders`: `Authorization` header — не утекает ли API key в логи/кэш?
- [ ] `_fetchModelsJson`: `AbortSignal.timeout(10_000)` — работает ли во всех браузерах?
- [ ] `loadModels()`: Cherry пагинация — guard `50` итераций, что если `total` бесконечный?

### Streaming
- [ ] `_parseSSE`: `buffer.split('\n\n')` — что если `data:` строка разбита на два chunk'а?
- [ ] `_parseSSE`: `[DONE]` — прерывает ли чтение `reader`? Есть ли `reader.cancel()`?
- [ ] `_parseNDJSON`: `json.done` — всегда ли Ollama отправляет `done: true`?
- [ ] `_parseSSE`: Anthropic `content_block_start` — корректно ли обрабатывается?
- [ ] `_extractResponsesContent`: OpenAI Responses API — покрыты ли все варианты формата?

### Cache
- [ ] `LLMCache.get()`: TTL вычисляется каждый раз — не тормозит ли на большом кэше?
- [ ] `LLMCache.set()`: `while (_data.order.length >= max)` — O(n) удаление, проблема при max=200?
- [ ] `LLMCache.invalidate()`: сбрасывает `_data = null` — безопасно ли при параллельных вызовах?
- [ ] Кэш ключ: `profileId:model:thinkingMode:cacheKey` — что если модель содержит `:`?

### Security
- [ ] `_esc()`: экранирует `& < > " '` — достаточно ли для XSS в innerHTML?
- [ ] `getPrompt()`: `replaceAll('{' + k + '}', v)` — пользовательские `vars` — есть ли инъекция?
- [ ] `_Storage?.loadLLMKey?.(profileId)` — API key хранится в localStorage, безопасно ли?
- [ ] `LLMSettingsModal._esc()`: используется ли для вывода пользовательских данных?

### Edge Cases
- [ ] `request()` с пустым `messages` — что происходит?
- [ ] `request()` с `profile.model = ''` — как обрабатывается?
- [ ] `_buildRequestBody`: `temperature: undefined` — отправляется ли в JSON?
- [ ] `_withNoThink`: если все messages пустые — корректно ли добавляет `/no_think`?
- [ ] `estimateTokens()`: `Math.ceil(length / 4)` — насколько точна для CJK/эмодзи?
- [ ] `hashStr()`: `Math.random().toString(36)` — collision при большом количестве вызовов?

### Race Conditions
- [ ] `_sem.acquire()` + `_localSem.acquire()` — что если local Sem освобождается до global?
- [ ] `LLMCache._load()` lazy init — при параллельных `get()`/`set()` возможен ли гонка?
- [ ] `LLMRequestLog.append()` — `_log.unshift()` + `_log.length = limit` — safe при параллельных вызовах?

### LLMSettingsModal (UI)
- [ ] `_armDangerButton()`: `setTimeout` 2.5s — что если компонент уничтожен до истечения?
- [ ] `_bindEvents()`: event listeners — корректно ли снимаются при `destroy()`?
- [ ] XSS через `innerHTML` — все ли данные проходят через `_esc()`?
- [ ] `_promptCollapsedGroups` — `Set` может ли утечь при частом открытии/закрытии модала?

## Пример хорошего тикета

```
[high] Retry counter не сбрасывается при успешном ответе
- Строка: 613-658
- Описание: `retries` декрементируется при ошибке, но не восстанавливается.
  При последовательных ошибках (429 → ok → 429) на второй серии ошибок
  будет меньше попыток, чем ожидалось.
- Влияние: При нестабильном соединении LLM-запросы могут завершаться
  раньше, чем нужно.
- Рекомендация: `retries` должен быть `const`, а текущий retry count
  храниться в отдельной переменной, сбрасываемой после успешного ответа.
```
