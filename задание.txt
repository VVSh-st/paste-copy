Ниже 2 анализа `word-complete.js` + доп анализ анализа и итог.
Анализаторы не видели весь проект и в некоторых местах могут ошибаться, проведи расследование, что хорошо и полезно для нашего проекта.


# Анализ `word-complete.js`

Я изучил весь файл. Это хорошо структурированный модуль автодополнения с приятным «matrix»-эффектом. Код в целом качественный, но есть ряд проблем — от потенциальных багов до архитектурных улучшений. Разберу по модулям.

---

## 🔴 Критические проблемы

### 1. `WordDict.build()` — несогласованность частот `_dynWords`

```javascript
for (const [w, freq] of _dynWords) {
  if (!_staticWords.has(w))
    wordFreq.set(w, (wordFreq.get(w) || 0) + freq);
}
// ... позже ...
words.forEach((w, i) => {
  wordFreq.set(w, (wordFreq.get(w) || 0) + mult);   // ← двойной учёт!
```

**Проблема:** слово из `_dynWords` сначала вносится в `wordFreq`, а затем при проходе по текстам **прибавляется ещё раз**. Динамические слова получают завышенный вес (по сути дублируются: накопленная частота + текущий проход).

**Как исправить** — динамические слова нужны только как «память» о словах, которых уже нет в текущих текстах. Внесите их **после** прохода по текстам, только если слова там нет:

```javascript
function build() {
  wordFreq.clear();
  bigrams.clear();
  _seedFromStatic();

  const activeId = State.getActive()?.id;
  const sessionCounts = new Map();

  State.getAll().forEach(tab => {
    const mult  = tab.id === activeId ? 2 : 1;
    const texts = [];
    collectTexts(tab.blocks, texts);
    texts.forEach(text => {
      const words = tokenize(text);
      words.forEach((w, i) => {
        wordFreq.set(w, (wordFreq.get(w) || 0) + mult);
        if (!_staticWords.has(w) && w.length >= cfg.minLen)
          sessionCounts.set(w, (sessionCounts.get(w) || 0) + mult);
        if (i > 0) {
          const prev = words[i - 1];
          if (!bigrams.has(prev)) bigrams.set(prev, new Map());
          const bm = bigrams.get(prev);
          bm.set(w, (bm.get(w) || 0) + mult);
        }
      });
    });
  });

  // Динамические слова добавляем ТОЛЬКО если их нет в текущих текстах
  for (const [w, freq] of _dynWords) {
    if (!_staticWords.has(w) && !wordFreq.has(w))
      wordFreq.set(w, freq);
  }

  for (const [w, cnt] of sessionCounts)
    _dynWords.set(w, Math.max(_dynWords.get(w) || 0, cnt));
}
```

### 2. `InlineHint` / `WordAcceptEffect` — обращение к `document.body` на этапе IIFE

```javascript
document.body.appendChild(mirrorEl);   // упадёт, если body ещё нет
```

В коде есть комментарий-требование «загружать с defer». Но если требование нарушат — будет «тихая» поломка `Cannot read properties of null`. 

**Рекомендация:** добавьте защитную инициализацию или явную ошибку:

```javascript
if (!document.body) {
  throw new Error('[InlineHint] требуется defer или подключение в конце <body>');
}
```

Лучше — ленивая инициализация DOM-узлов при первом `show()`.

---

### 3. `WordDict` — глобальные зависимости `State` без проверки

`build()` напрямую вызывает `State.getActive()`, `State.getAll()`. Если `State` ещё не загружен (порядок скриптов), `loadWordlist()` → `build()` упадёт.

**Рекомендация:** защита:

```javascript
function build() {
  if (typeof State === 'undefined' || !State?.getAll) return;
  // ...
}
```

## 🟡 Логические и UX-улучшения

### 4. `suggest()` — линейный перебор всего словаря O(n)

```javascript
for (const [w, freq] of wordFreq) {
  if (w.length > p.length && w.startsWith(p)) ...
}
```

На каждый ввод символа перебирается **весь** словарь. При больших словарях (10k+ слов) это заметная задержка на каждое нажатие.

**Рекомендация:** для серьёзного словаря постройте префиксный индекс (trie) или хотя бы индекс по первым 2-3 буквам:

```javascript
// Индекс: первые 2 символа → массив слов
const prefixIndex = new Map();

function rebuildIndex() {
  prefixIndex.clear();
  for (const w of wordFreq.keys()) {
    const key = w.slice(0, 2);
    if (!prefixIndex.has(key)) prefixIndex.set(key, []);
    prefixIndex.get(key).push(w);
  }
}

function suggest(prefix) {
  if (!cfg.enabled || prefix.length < cfg.minLen) return [];
  const p = prefix.toLowerCase();
  const bucket = prefixIndex.get(p.slice(0, 2)) || [];
  const out = [];
  for (const w of bucket) {
    if (w.length > p.length && w.startsWith(p))
      out.push({ w, effective: wordFreq.get(w) + (_staticWords.has(w) ? STATIC_BOOST : 0) });
  }
  out.sort((a, b) => b.effective - a.effective);
  return out.slice(0, cfg.maxSuggest).map(x => x.w);
}
```
(Индекс перестраивать в конце `build()`.)

---

### 5. `suggestNext()` — нет учёта префикса и минимальной частоты

Биграмма всегда возвращает самое частое следующее слово, даже если оно встретилось всего 1 раз. Это даёт «шумные» подсказки.

**Рекомендация:** добавьте порог:

```javascript
function suggestNext(prevWord, minCount = 2) {
  if (!cfg.enabled || !prevWord) return null;
  const bm = bigrams.get(prevWord.toLowerCase());
  if (!bm?.size) return null;
  let best = null, max = minCount - 1;
  for (const [w, c] of bm) if (c > max) { best = w; max = c; }
  return best;
}
```

### 6. `WordComplete.handleKeydown` — `State.snapshot()` без проверки

```javascript
State.snapshot();
WordDict.scheduleBuild();
```

Те же глобальные зависимости. Если модуль переиспользуется в другом контексте — упадёт. Стоит обернуть в проверки или внедрять зависимости через init().

### 7. Слушатели `scroll`/`resize` скрывают подсказку всегда

```javascript
window.addEventListener('scroll', hide, { passive: true, capture: true });
```

Любой скролл (даже в другом элементе) убивает подсказку. Часто это раздражает. Можно вместо `hide()` **перепозиционировать** подсказку, если активная textarea ещё в фокусе:

```javascript
function reposition() {
  if (_visible && activeTa) show(activeTa, suffix);
  else hide();
}
window.addEventListener('scroll', reposition, { passive: true, capture: true });
```

(С throttle через `requestAnimationFrame`.)

## 🟢 Мелкие замечания / чистота кода

### 8. `tokenize()` — апострофы в середине слов
Регэксп `[^\p{L}\p{N}']+` оставляет апострофы. Слово `don't` → `don't`, но `''abc''` чистится только по краям. Для русского это редко важно, но `'` внутри может зашумлять словарь. Проверьте, нужен ли апостроф вообще.

### 9. `_buildTempPayload()` дублирует фильтр с `_loadTempWordlist()`
Условие `freq >= 2` и `!_staticWords.has(w)` повторяется в нескольких местах. Вынесите в функцию:

```javascript
const _isPersistable = (w, freq) =>
  !_staticWords.has(w) && w.length >= cfg.minLen &&
  typeof freq === 'number' && freq >= 2;
```

### 10. `WordAcceptEffect` — `measureCanvas` создаётся, но `getContext` может вернуть null
Есть проверка `if (!measureCtx ...)` — хорошо. Но `measureCanvas` не нужен в DOM, всё ок.

### 11. Magic numbers
`STATIC_BOOST = 10000`, `34` (мс кадра), `0.62`, `0.58` в `revealGlyph` — стоит вынести в именованные константы с комментариями для будущей настройки.

### 12. `getConfig()` возвращает мутабельную ссылку
```javascript
function getConfig() { return cfg; }
```
Внешний код может напрямую мутировать `cfg`, минуя `setConfig`. Верните копию:
```javascript
function getConfig() { return { ...cfg }; }
```

# 1. Общая оценка

Файл реализует несколько независимых модулей:

1. `WordDict`  
   Частотный словарь, биграммы, загрузка `wordlist.json`, временный пользовательский словарь, автосохранение.

2. `InlineHint`  
   Отрисовка ghost-подсказки рядом с кареткой в `textarea`.

3. `WordAcceptEffect`  
   Визуальный эффект принятия подсказки.

4. `WordComplete`  
   Логика показа подсказки и принятия по `Tab`.

5. `SmartList`  
   Автонумерация списков при `Enter`.

Код в целом аккуратно разделён на модули, использует IIFE, не засоряет глобальную область сверх необходимых `window.WordComplete`, `window.WordDict` и т.д. Видно, что учтены IME-композиция, CORS для `file://`, fallback для File System Access API, `prefers-reduced-motion`.

Но есть несколько важных проблем:

- возможные ошибки при загрузке до готовности `document.body`;
- зависимость от глобального `State`;
- не до конца корректная логика динамического словаря;
- потенциальные проблемы с производительностью на больших текстах;
- неполная синхронизация mirror-элемента для `textarea`;
- риск неправильного позиционирования подсказки;
- слабая обработка удаления/изменения пользовательских слов;
- устаревание подсказки между `keydown` и `Tab`;
- сохранение временного словаря только раз в 2 часа и при `beforeunload`, что не всегда надёжно.

# 2. `WordDict`

## 2.1. Сильные стороны

Хорошо:

```js
const wordFreq = new Map();
const bigrams  = new Map();
```

Использование `Map` удобно для частот.

Хорошо, что статические слова и динамические разделены:

```js
const _staticWords   = new Map();
const _staticBigrams = new Map();
const _dynWords = new Map();
```

Также правильно, что статическим словам даётся boost:

```js
const STATIC_BOOST = 10000;
```

Это позволяет словам из `wordlist.json` иметь приоритет.

---

## 2.2. Проблема: `State` используется без проверки

В `build()`:

```js
const activeId = State.getActive()?.id;

State.getAll().forEach(tab => {
```

Если `State` ещё не загружен или не существует, скрипт упадёт.

### Что улучшить

Добавить защиту:

```js
function _getStateSafe() {
  if (!window.State || typeof State.getAll !== 'function') {
    return null;
  }
  return State;
}
```

И в `build()`:

```js
function build() {
  wordFreq.clear();
  bigrams.clear();
  _seedFromStatic();

  for (const [w, freq] of _dynWords) {
    if (!_staticWords.has(w)) {
      wordFreq.set(w, (wordFreq.get(w) || 0) + freq);
    }
  }

  const state = _getStateSafe();
  if (!state) return;

  const activeId = state.getActive?.()?.id;

  state.getAll().forEach(tab => {
    // ...
  });
}
```

Это особенно важно, если файл подключается раньше других модулей.

---

## 2.3. Проблема: динамические слова не уменьшаются после удаления текста

В `build()` текущая логика:

```js
_dynWords.set(w, Math.max(_dynWords.get(w) || 0, cnt));
```

То есть если слово когда-то встретилось 10 раз, а потом пользователь удалил весь текст, слово останется в `_dynWords` с частотой 10.

Комментарий говорит:

```js
// чтобы частоты отражали реальную встречаемость, а не суммировались бесконечно.
```

Но по факту это отражает максимальную историческую встречаемость, а не текущую.

Это может быть намеренно, если `_dynWords` — обученный пользовательский словарь. Но тогда нужно понимать последствия:

- случайно введённые слова останутся;
- опечатки будут сохраняться;
- удалённые или устаревшие термины будут продолжать предлагаться.

### Вариант 1. Оставить как есть, но добавить затухание

Можно хранить не только частоту, но и дату обновления:

```json
{
  "words": [
    ["example", 5, "2026-06-27T12:00:00.000Z"]
  ]
}
```

И периодически снижать вес старых слов.

Например:

```js
const DYN_DECAY_DAYS = 30;
```

### Вариант 2. Разделить текущий словарь и пользовательскую память

Например:

```js
const _sessionWords = new Map();
const _userMemoryWords = new Map();
```

`wordFreq` строить как:

```js
static + currentState + userMemory * 0.5
```

Тогда опечатки из прошлого будут иметь меньший вес.

---

## 2.4. Проблема: биграммы не сохраняются во временный словарь

Сейчас `_buildTempPayload()` сохраняет только:

```js
words
```

Но не сохраняет `bigrams`.

При следующей сессии будут восстановлены только слова, но не связи между словами.

### Что улучшить

Если подсказки следующего слова важны, сохраняйте динамические биграммы.

Например:

```js
const _dynBigrams = new Map();
```

В `build()` собирать `sessionBigrams`, затем обновлять `_dynBigrams`.

Формат:

```json
{
  "meta": {},
  "words": [
    ["hello", 5]
  ],
  "bigrams": {
    "hello": {
      "world": 3
    }
  }
}
```

Загрузка:

```js
if (data.bigrams && typeof data.bigrams === 'object') {
  for (const [prev, followers] of Object.entries(data.bigrams)) {
    // ...
  }
}
```

## 2.5. Проблема: `suggest()` линейно перебирает весь словарь

```js
for (const [w, freq] of wordFreq) {
  if (w.length > p.length && w.startsWith(p))
    out.push(...)
}
```

Для маленького словаря нормально. Для 10–50 тысяч слов уже может быть заметно, особенно при каждом вводе символа.

### Что улучшить

#### Вариант простой: кэшировать результаты

```js
const suggestCache = new Map();

function suggest(prefix) {
  if (!cfg.enabled || !prefix || prefix.length < cfg.minLen) return [];

  const p = prefix.toLowerCase();

  const cacheKey = `${p}|${cfg.maxSuggest}`;
  if (suggestCache.has(cacheKey)) return suggestCache.get(cacheKey);

  const out = [];

  for (const [w, freq] of wordFreq) {
    if (w.length > p.length && w.startsWith(p)) {
      out.push({
        w,
        effective: freq + (_staticWords.has(w) ? STATIC_BOOST : 0)
      });
    }
  }

  out.sort((a, b) => b.effective - a.effective);

  const result = out.slice(0, cfg.maxSuggest).map(x => x.w);
  suggestCache.set(cacheKey, result);
  return result;
}
```

И очищать кэш в `build()`:

```js
suggestCache.clear();
```

#### Вариант лучше: индекс по первым буквам

Например:

```js
const prefixIndex = new Map();
```

Строить индекс для первых 2–3 символов. Тогда `suggest('com')` перебирает только слова, начинающиеся на `co`, а не весь словарь.

## 2.6. Проблема: `loadWordlist()` при неудачном HTTP не вызывает `build()`

Сейчас:

```js
if (!r.ok) {
  console.debug(`[WordDict] wordlist.json not found (HTTP ${r.status})`);
  return;
}
```

Но поскольку есть `finally`, `_loadTempWordlist(); build();` всё равно выполнится.

Это нормально.

Однако если `fetch` успешен, но JSON невалидный, ошибка будет обработана в `catch`, затем `finally` также выполнится.

Тоже нормально.

---

## 2.7. Проблема: при `file://` `loadWordlist()` вызывает `_loadTempWordlist()` и `build()`, затем выходит

```js
if (location.protocol === 'file:') {
  _loadTempWordlist();
  build();
  return;
}
```

Это хорошо. Но в этом режиме статический `wordlist.json` никогда не загрузится.

Если проект часто запускается локально как файл, стоит добавить ручной импорт словаря или предупреждение в UI.

---

## 2.8. Проблема: `beforeunload` с async/синхронностью

```js
window.addEventListener('beforeunload', () => _saveTempToLocalStorage());
```

Сохранение в `localStorage` синхронное, это нормально.

Но запись в файл через `_fsHandle` при закрытии не выполняется. Если пользователь рассчитывает на файл, он может потерять свежие данные между автосохранениями.

### Что улучшить

Добавить сохранение чаще, например debounce после изменений.

```js
let _dirty = false;
let _saveDebounce = null;

function markDirty() {
  _dirty = true;
  clearTimeout(_saveDebounce);
  _saveDebounce = setTimeout(() => {
    const payload = _buildTempPayload();
    if (!payload) return;
    _saveTempToLocalStorage(payload);
    _tryWriteToHandle(payload);
    _dirty = false;
  }, 10000);
}
```

И вызывать `markDirty()` после `build()` или после принятия подсказки.

# 3. Токенизация

Текущая функция:

```js
function tokenize(text) {
  if (!text) return [];
  return text.split(/[^\p{L}\p{N}']+/u)
    .map(w => w.toLowerCase().replace(/^'+|'+$/g, ''))
    .filter(w => w.length >= 2);
}
```

## 3.1. Потенциальная проблема с апострофом

Слова вроде:

- `don't`
- `it's`
- `l'homme`

будут сохранены нормально.

Но одиночные комбинации или странные формы тоже попадут.

## 3.2. Потенциальная проблема с дефисными словами

Слова типа:

- `по-русски`
- `что-то`
- `state-of-the-art`

разобьются на части.

Если это нежелательно, можно включить дефис внутрь токена:

```js
return text
  .split(/[^\p{L}\p{N}'-]+/u)
  .map(w => w.toLowerCase().replace(/^['-]+|['-]+$/g, ''))
  .filter(w => w.length >= 2);
```

Но тогда нужно осторожно с тире и пунктуацией.

# 4. `InlineHint`

## 4.1. Сильные стороны

Хороший подход: mirror-элемент для вычисления координат каретки.

```js
const mirrorEl = document.createElement('div');
```

Это стандартный способ позиционирования подсказки в `textarea`.

---

## 4.2. Важная проблема: mirror не копирует border и height

Вы вычисляете координаты так:

```js
const caret = getCaretXY(ta, ta.selectionStart);
const insertX = Math.round(rect.left + bordL + caret.x);
const rawY  = Math.round(rect.top  + bordT + caret.y);
```

Но mirror имеет:

```js
border: 'none',
margin: '0',
```

И копирует только часть стилей. Для точного совпадения нужно особенно учитывать:

- width;
- padding;
- font;
- line-height;
- white-space;
- word-wrap;
- box-sizing;
- border;
- scroll;
- textarea wrapping.

Сейчас `border` не копируется, но добавляется `bordL`, `bordT` отдельно. Это может быть нормально.

Однако mirror width:

```js
mirrorEl.style.width = ta.clientWidth + 'px';
```

Если `boxSizing: border-box`, а padding скопирован, может получиться несовпадение. `clientWidth` включает padding, но не border. Для `box-sizing: border-box` это может исказить переносы.

### Что улучшить

Обычно для mirror используют:

```js
mirrorEl.style.width = ta.clientWidth + 'px';
mirrorEl.style.height = ta.clientHeight + 'px';
```

И дополнительно:

```js
mirrorEl.style.overflow = 'hidden';
```

```js
'borderTopWidth', 'borderRightWidth', 'borderBottomWidth', 'borderLeftWidth',

'textAlign',
'direction',
'writingMode',
```

Особенно важны:

```js
'textAlign',
```

## 4.3. Проблема: marker имеет нулевую высоту

```js
marker.style.cssText =
```

Иногда marker с нулевой высотой может давать странный `top`.

Часто используют символ-заглушку:

```js

```

Или:


Рекомендованный вариант:

```js
marker.textContent = '\u200b';
```

## 4.4. Проблема: подсказка не обновляется при прокрутке внутри textarea

`show()` вычисляет положение на момент вызова. Если пользователь скроллит textarea, подсказка может остаться на месте.

Есть общий обработчик:

```js
window.addEventListener('scroll', hide, { passive: true, capture: true });
```

Но прокрутка внутри `textarea` не всегда всплывает как window scroll.

### Что улучшить

Добавить обработчик на активную textarea:

```js
function show(ta, suf) {
  // ...
  if (activeTa && activeTa !== ta) {
    activeTa.removeEventListener('scroll', hide);
  }

  activeTa = ta;
  ta.addEventListener('scroll', hide, { passive: true });
}
```

В `hide()`:

```js
if (activeTa) {
  activeTa.removeEventListener('scroll', hide);
}
```

Но аккуратно, чтобы не удалять не тот listener.

## 5.4. Проблема: blur удаляет элементы, но не чистит WeakMap

## 6.2. Потенциальная проблема: `Tab` принимает устаревшую подсказку

Сценарий:

1. Подсказка показана.
2. Пользователь изменил текст способом, который не вызвал `handleInput`.
3. Нажал `Tab`.
4. Старый suffix вставился в новый контекст.

Перед вставкой лучше перепроверять, что подсказка всё ещё соответствует текущему слову.

### Улучшение

В `handleKeydown()` перед принятием:

```js
const { currentWord } = getContext(ta);
const cfg = WordDict.getConfig();

if (currentWord.length >= cfg.minLen) {
  const candidates = WordDict.suggest(currentWord);
  if (!candidates.length || !candidates[0].startsWith(currentWord)) {
    InlineHint.hide();
    return;
  }

  const expectedSuffix = candidates[0].slice(currentWord.length);
  if (expectedSuffix !== suf) {
    InlineHint.hide();
    InlineHint.show(ta, expectedSuffix);
    return;
  }
}
```

Для биграммы тоже можно проверить:

```js
if (!currentWord) {
  const { prevWord } = getContext(ta);
  const next = WordDict.suggestNext(prevWord);
  if (next !== suf) {
    InlineHint.hide();
    return;
  }
}
```

# 8. Инициализация и зависимость от DOM

В нескольких местах код сразу делает:

```js
document.body.appendChild(...)
```

Например:

```js
document.body.appendChild(mirrorEl);
document.body.appendChild(hintEl);
```

В комментариях написано:

```js
// Требование: скрипт должен загружаться с defer или в конце <body>
```

Это рабочее требование, но хрупкое.

### Что улучшить

Сделать безопасный DOM-ready helper.

```js
function onDomReady(fn) {
  if (document.readyState === 'loading') {
    document.addEventListener('DOMContentLoaded', fn, { once: true });
  } else {
    fn();
  }
}
```

Но здесь модули сразу создают DOM-элементы при IIFE. Чтобы сделать правильно, нужно либо:

1. создать `init()` у каждого DOM-зависимого модуля;
2. вызывать `init()` после DOM ready.

Например для `InlineHint`:

```js
const InlineHint = (() => {
  let mirrorEl = null;
  let hintEl = null;

  function init() {
    if (mirrorEl) return;

    mirrorEl = document.createElement('div');
    // ...
    document.body.appendChild(mirrorEl);

    hintEl = document.createElement('div');
    // ...
    document.body.appendChild(hintEl);
  }

  function show(ta, suf) {
    init();
    // ...
  }

  onDomReady(init);

  return { init, show, hide, ... };
})();
```

Это позволит подключать файл где угодно.

## 12.4. Кэш для `suggest()`

В `WordDict` добавить:

```js
const suggestCache = new Map();
```

В `build()` в начале или конце:

```js

```

Изменить `suggest()`:

```js
function suggest(prefix) {
  if (!cfg.enabled || !prefix || prefix.length < cfg.minLen) return [];

  const p = prefix.toLowerCase();
  const cacheKey = `${p}|${cfg.maxSuggest}`;

  if (suggestCache.has(cacheKey)) {
    return suggestCache.get(cacheKey);
  }

  const out = [];

  for (const [w, freq] of wordFreq) {
    if (w.length > p.length && w.startsWith(p)) {
      out.push({
        w,
        effective: freq + (_staticWords.has(w) ? STATIC_BOOST : 0)
      });
    }
  }

  out.sort((a, b) => b.effective - a.effective);

  const result = out.slice(0, cfg.maxSuggest).map(x => x.w);
  suggestCache.set(cacheKey, result);

  return result;
}
```

Ревью качественное и подробное. Многое пересекается с моим анализом (защита `State`, кэш/индекс для `suggest`, DOM-ready, биграммы). Но есть **важные нюансы**, где другая нейросеть права сильнее меня, а есть пара мест, где она ошибается или упускает главное. Разберу по пунктам.

```javascript
lineHeight: (parseFloat(snapshot.lineHeight) - 1) + 'px',
```
Если `getComputedStyle` вернёт `lineHeight: 'normal'` (что бывает, если в CSS явно не задан line-height), то `parseFloat('normal')` → `NaN` → `'NaNpx'`. Эффект сломается визуально. Патч корректный, его стоит внести **обязательно**.

⚠️ Уточнение: на практике `getComputedStyle().lineHeight` для большинства textarea возвращает пиксельное значение, а не `normal`. Но «normal» возможен — защита оправдана.

### 2. `InputEvent` с `bubbles: true` (п. 6.3) — **верное и важное**
```javascript
ta.dispatchEvent(new Event('input'));
```
`new Event('input')` **не всплывает** (`bubbles: false` по умолчанию). Если в проекте есть делегирование `input` на родителе — оно не сработает. Это реальный потенциальный баг. Helper с fallback на `InputEvent` — правильное решение.

⚠️ Но: нужно проверить, **как именно** в проекте слушается `input`. Если везде слушатель вешается прямо на textarea — проблема не проявляется. Перед внесением проверьте остальной код проекта.

### 3. Контекст URL/email (п. 13.3) — **хорошая UX-идея**, которую я не упомянул
Подсказки внутри `https://exa...` действительно раздражают. Дешёвая и полезная правка.

### 4. Прокрутка **внутри** textarea (п. 4.4) — точное уточнение
Я предлагал перепозиционирование на window-scroll, но ревью верно замечает: scroll внутри textarea **не всплывает** до window. Нужен отдельный listener на самой textarea.

## ⚠️ Где ревью ошибается или вводит в заблуждение

### 1. Кэш `suggest()` (п. 2.5 / 12.4) — **сомнительно, лучше мой вариант с индексом**
Предложенный кэш `Map` по ключу `prefix|maxSuggest`:
```javascript
const suggestCache = new Map();
suggestCache.set(cacheKey, result);
```
**Проблемы:**
- Кэш **не ограничен по размеру** — при активном вводе он растёт неограниченно (по сути memory leak до следующего `build()`).
- Кэш **не ускоряет первый ввод** каждого префикса — а именно первый ввод обычно и тормозит, т.к. пользователь печатает новые буквы.
- Промахи кэша при печати по букве почти 100% (каждый новый символ = новый ключ).

**Мой вариант (префиксный индекс по первым 2 буквам)** решает корень проблемы — он сокращает O(n) до O(размер бакета) для **любого** ввода, включая первый. Это объективно лучше для данной задачи.

Если уж делать кэш — то с лимитом (LRU) **поверх** индекса, не вместо.

### 2. Динамический словарь / `Math.max` (п. 2.3) — **ревью НЕ заметило главный баг**
Ревью корректно обсуждает «затухание» и «опечатки навсегда», но **полностью пропустило двойной учёт частот**, который я нашёл:
```javascript
for (const [w, freq] of _dynWords) {        // +freq
  if (!_staticWords.has(w))
    wordFreq.set(w, (wordFreq.get(w) || 0) + freq);
}
// ...
words.forEach((w, i) => {
  wordFreq.set(w, (wordFreq.get(w) || 0) + mult);  // ← ещё раз!
```
Это конкретный **баг в текущем коде** (слово учитывается дважды за один build), а не философия о старении. Мой п. 1 важнее их п. 2.3 — он чинит реальное искажение весов прямо сейчас.

### 3. `beforeunload` + debounce-сохранение (п. 2.8) — спорно
Идея `markDirty` с debounce 10 сек разумна, но ревью предлагает вызывать `_tryWriteToHandle` из debounce — а это **async запись в файл**, которая в `beforeunload` не успеет завершиться. Сам по себе debounce полезен, но не решает проблему потери данных при закрытии (запись в файл всё равно асинхронна). Для localStorage достаточно текущего `beforeunload`. Это улучшение «средней» полезности, не критичное.

### 4. mirror: добавить `writingMode`, `direction` и border-стили (п. 4.2 / 12.x)
Само по себе верно для RTL/вертикального текста. Но:
- Для **русско-английского** интерфейса (что видно по коду) RTL почти не нужен — приоритет низкий.
- Добавление `borderTopWidth` и т.п. в `MIRROR_PROPS` при том, что код **уже** компенсирует border отдельно через `bordL`/`bordT`, может привести к **двойному** учёту border и сдвигу подсказки. Здесь ревью невнимательно — нужно либо border в mirror, либо ручная компенсация, **не оба сразу**.

- **`getConfig()` возвращает мутабельную ссылку** — я отметил, ревью пропустило. Реальный риск нарушения инкапсуляции.
- **Throttle перепозиционирования** через `requestAnimationFrame` — оба упомянули перепозиционирование, но без throttle частые scroll-события будут дёргать layout.
