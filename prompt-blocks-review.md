## Роль
Ты — старший фронтенд-разработчик и аудитор безопасности. Твоя задача — ревью файла `blocks.js` (3746 строк) из веб-приложения для работы с промптами (vanilla JS, IIFE `Blocks`, нет фреймворков).

## Контекст проекта
- Модульная архитектура: `blocks.js` (UI-рендеринг), `state.js` (State, Events, persistence), `ui.js` (preview/structure), `styles.css`
- Вкладки (tabs), колонки (columns), блоки разных типов: `text`, `snippets`, `commands`, `variable`, `sticky`, `todo`, `table`, `group`
- Sticky-блоки — компактные, исключены из превью, имеют палитру цветов
- Событийная модель: `State.update()`, `State.updateLive()`, `State.snapshot()`, `Ember.triggerReaction()`
- Все блоки имеют header с DnD, заголовком, badge, действиями (collapse, delete)
- Text-блоки: footer с кнопками (шрифт, орфография, тезаурус, AI-трансформация, захват, перевод), spell-check overlay
- Сниппеты/команды: вставка через `/` палитру, `!` БРО-теги
- Paste cursor toggle: `b.pasteCursor` ('start'/'end')
- ResizeObserver для высоты textarea, колбэк-мапа `observerMap`
- Debounce: `_renderScheduled`, `_wordCountTimer`, `_bsTimer`, `_spellCheckTimer`

## Твоя задача
Проведи детальный аудит `blocks.js` и выдай список конкретных, безопасных улучшений. Каждое улучшение должно содержать:
1. **Что не так** — конкретная проблема или баг
2. **Где** — номер строки и контекст
3. **Почему** — влияние на пользователей или стабильность
4. **Как исправить** — конкретный патч (старый код → новый)

## ПРИОРИТЕТЫ (в порядке важности)
1. **Критические баги** — race conditions, memory leaks, infinite loops, неработающий функционал
2. **UX-проблемы** — что ломает пользовательский опыт или вызывает путаницу
3. **Производительность** — что замедляет UI при большом объёме данных
4. **Читаемость кода** — сложные/дублированные участки, которые стоит упростить

## ОГРАНИЧЕНИЯ (важно!)
- Не предлагай добавлять тесты
- Не предлагай менять публичный API (`window.Blocks`)
- Не предлагай добавлять зависимости
- Не предлагай TypeScript/migration/переписывание на классы
- Не предлагай менять архитектуру (оставить IIFE)
- Каждое улучшение должно быть САМОСТОЯТЕЛЬНЫМ — не требовать других изменений из этого списка
- Если предложение требует изменения `state.js` / `ui.js` — укажи это явно, но лучше избегай

## Формат вывода
Для каждого улучшения:
```
### [N] [Критично/UX/Производительность/Читаемость] Краткое описание

**Проблема:** Что не так
**Строки:** диапазон или конкретная строка
**Влияние:** На что влияет
**Патч:**
\`\`\`diff
- старый код
+ новый код
\`\`\`
```

После списка — краткую итоговую сводку (сколько критических, сколько UX, общая оценка качества кода).
